# MSA란?
## Microservices Architecture (MSA)
- 하나의 애플리케이션을 여러개의 독립적인 서비스로 분리하여 개발, 배포, 유지, 보수를 용이하게 하는 아키텍처 스타일입니다.
- 각 서비스는 특정 비즈니스 기능을 수행하며, 서로 독립적으로 배포되고 확장될 수 있습니다.
	- 각 서비스는 서로 다른 서비스에 영향을 주지 않습니다.
	- 각 서비스 별로 적절한 기술 스택을 자유롭게 선택할 수 있습니다.
- 각 서비스는 주로 HTTP/HTTPS, 메시지 큐 등을 통해 통신합니다.

## 모놀리틱 아키텍처와의 비교
### 모놀리틱
- 하나의 큰 코드 베이스로 구성된 애플리케이션으로, 모든 기능이 하나의 애플리케이션 내에 포함된다.
**장점**
- 하나의 코드 베이스로 구서어되어 배포가 단순하다.
- 하나의 데이터베이스를 사용해 ==데이터 일관성을 쉽게 유지==할 수 있다.
**단점**
- ==특정 기능을 확장==하려고 하거나 ==작은 변경==을 시도하려면 ==전체 애플리케이션을 확장/배포==해야한다.
- 새로운 기술 도입이 어렵고, 특정 모듈에 종속적이다.
### MSA
**장점**
- 특정 서비스만 확장하여 성능을 최적화 할 수 있다.
- 개별 서비스의 변경사항을 독립적으로 배포 가능하여 배포 주기를 단축 할 수 있다.
- 서비스별로 적합한 기술 스택을 선택할 수 있다.

**단점**
- 서비스간 통신, 데이터 일관성 유지 등의 ==복잡성이 증가==한다
- 각 서비스의 모니터링, 로깅 등을 개별적으로 관리함으로 ==운영비용이 증가==한다.
- 분산된 데이터베이스로 인해 데이터 일관성 유지가 어려울 수 있다.
- 서비스 간의 통신으로 인해 지연시간이 발생할 수 있다.

> 신규 프로젝트의 경우 빠른 랜딩이 중요하기에 모놀리틱이 나을 수 있다.
